# ADR 0004: Profile-centric config with derived artifacts (lockfile pattern)

- Status: Accepted
- Date: 2026-05-22

## Context

Slices 1-4 left rate-limit config fragmented across three places: `lib/rate_limiter.py:DAILY_LIMITS` (legacy dict), `lib/engagement/policy.py:EngagementPolicy` (slice 1 quotas), and test expectations. A slice 4 audit uncovered a three-way mismatch on Instagram comment quota: `DAILY_LIMITS = 7`, `EngagementPolicy = 10`, test expects `2`. Schedule config is similarly scattered (five files: `schedule.json`, `~/Library/LaunchAgents/` plists, stale templates, CLAUDE.md skills catalog, `publishing_timeline.json`). Adding a new platform (Threads, Bluesky) required edits in 6+ places. Naming was also inconsistent (`instagram:ig_comment` legacy prefix vs flat `<action>` keys).

## Decision

Canonical platform config lives in `social-automation/profiles/<platform>.json` (engine defaults). Brand-specific overrides live in `<brand_dir>/brand.json` (slice C; gitignored). Derived artifacts (`data/rate_limits.json` this slice; `schedule.json` slice B; plists slice D) are regenerated by `python -m tools.profiles_build` and committed (lockfile pattern, like `Cargo.lock`). CI guard via `--check` flag detects divergence. Naming cleanup: drop `ig_*` prefix; use flat `<platform>:<action>` keys across all platforms. Public name `DAILY_LIMITS` preserved as a derived view — existing callers untouched. `EngagementPolicy.from_profiles()` arrives alongside `from_config()` for opportunistic migration.

## Consequences

- (a) One source of truth per platform; bumping a rate limit is one file edit.
- (b) `DAILY_LIMITS` dict preserved as a derived artifact; no caller rewrites needed.
- (c) `EngagementPolicy.from_profiles()` factory method allows scanners to migrate at their own pace.
- (d) Future architecture reviews should NOT merge profile config back into `EngagementPolicy` or restore per-call rate-limit literals.
- (e) A new platform = one profile JSON + adapter implementation + brand-overlay entry. Today: 6+ files.
- (f) Slice B (schedule), C (brand overlay), D (plist generation) depend on this foundation.
